# Lab 009 - Microservice Split — Result

## Проверка

- [ ] Описаны границы сервисов.
- [ ] Для каждого сервиса указана ответственность.
- [ ] Зафиксированы контракты взаимодействия.
- [ ] Отдельно записаны риски и причины не дробить систему преждевременно.

## Итог

В ходе выполнения работы был разобран архитектурный переход от монолита к возможному микросервисному разделению TaskFlow backend. Основной результат — не код, а карта границ, зависимостей и рисков.

## Что стало понятнее

- Микросервисы начинаются с границ ответственности, а не с новых репозиториев.
- Данные должны иметь владельца.
- Сетевое взаимодействие добавляет новые классы ошибок.
