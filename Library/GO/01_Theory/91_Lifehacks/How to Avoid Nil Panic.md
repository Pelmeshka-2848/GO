# How to Avoid Nil Panic

## Суть лайфхака

Если значение может быть `nil`, это должно быть видно в контракте функции, проверке ошибки или названии.

## Когда использовать

- работаешь с pointer;
- возвращаешь найденный объект;
- используешь map/slice/channel;
- вызываешь методы у зависимостей;
- разбираешь panic `nil pointer dereference`.

## Прием

- Проверяй ошибку перед использованием результата.
- Для “не найдено” возвращай понятную ошибку или `(value, bool)`.
- Инициализируй map перед записью.
- Не вызывай метод на nil-зависимости.
- В тестах добавляй case для отсутствующего значения.

## Ограничения

- Не надо ставить nil-check везде подряд. Лучше сделать контракт ясным.
