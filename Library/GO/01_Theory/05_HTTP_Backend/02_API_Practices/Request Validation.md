# Request Validation

## Кратко

Request validation — проверка входных данных до выполнения бизнес-операции.

## Зачем нужно

Валидация защищает приложение от некорректных данных: пустых имен, неверных email, отрицательных чисел, слишком длинных строк, неправильных enum-значений.

## Где выполнять

Базовую проверку формата можно делать рядом с handler-ом. Бизнес-правила лучше держать в service layer.

## Правила

- Отличай синтаксическую валидацию от бизнес-валидации.
- Возвращай понятные ошибки клиенту.
- Не доверяй данным только потому, что клиент “должен” отправлять правильно.
- Проверяй обязательные поля.
- Не раскрывай внутренние детали валидации, если они чувствительные.

## Типичные ошибки

| Ошибка | Почему возникает | Как исправить |
|---|---|---|
| Нет проверки входа | Клиент “нормальный” | Валидировать на сервере |
| SQL constraint вместо API validation | Надежда на БД | Проверять раньше и понятнее |
| Разные форматы ошибок | Нет контракта | Единый response |
