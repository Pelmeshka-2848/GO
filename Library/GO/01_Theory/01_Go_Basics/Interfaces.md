# Interfaces

## Что это

Interface описывает поведение через набор методов.

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}
```

Тип реализует interface неявно: достаточно иметь нужные методы. Отдельно писать `implements` не нужно.

## Зачем нужно

Интерфейсы помогают отделять код от конкретной реализации:

- handler не обязан знать, какая база данных используется;
- сервис можно тестировать через fake-реализацию;
- разные реализации могут иметь один контракт поведения.

## Главное правило

Принимай interface там, где нужно поведение. Возвращай конкретный тип там, где создаешь реализацию.

## Маленькие интерфейсы

В Go ценятся маленькие интерфейсы:

```go
type UserFinder interface {
    FindUser(id int) (User, error)
}
```

Один-два метода часто лучше, чем большой универсальный interface.

## Ключевые правила

- Не создавай interface заранее “на будущее”.
- Interface должен отражать поведение, а не просто повторять все методы структуры.
- Лучше объявлять interface рядом с потребителем, а не с реализацией.
- Избегай `interface{}` или `any`, если можно описать конкретный тип.

## Частые ошибки

- делать огромные интерфейсы;
- создавать interface для каждой структуры автоматически;
- возвращать interface из конструктора без причины;
- путать nil interface и interface, внутри которого лежит nil pointer.
