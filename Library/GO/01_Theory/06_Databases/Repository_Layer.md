# Repository Layer

## Что это

Repository layer — слой, который отвечает за доступ к данным: SQL-запросы, чтение строк, запись сущностей, транзакции.

## Зачем нужно

Repository отделяет бизнес-логику от деталей хранения. Сервису не нужно знать, какой SQL нужен для получения пользователя или задачи.

## Ответственность repository

- выполнить SQL-запрос;
- передать параметры безопасно;
- преобразовать строки БД в Go-структуры;
- вернуть доменную ошибку или техническую ошибку;
- работать с транзакцией, если она передана.

## Что не должно быть в repository

- HTTP-статусы;
- парсинг JSON;
- бизнес-сценарии высокого уровня;
- логика авторизации;
- формат API-ответов.

## Ключевые правила

- Repository не должен зависеть от HTTP.
- Методы repository принимают `context.Context`.
- SQL должен быть параметризован.
- Ошибка “не найдено” должна быть отличима от внутренней ошибки.
- Транзакционные сценарии нужно проектировать явно.

## Частые ошибки

- писать SQL прямо в handler-е;
- возвращать `sql.ErrNoRows` наружу без нормализации;
- смешивать repository и service layer;
- использовать repository как свалку всей логики;
- прятать transaction management в неожиданных местах.
