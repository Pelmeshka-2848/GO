# Testing Rules

## Правило

Тест должен проверять наблюдаемое поведение кода, а не повторять его внутреннюю реализацию.

## Почему так

Если тест жестко привязан к внутренним деталям, любой рефакторинг ломает тесты, даже когда поведение осталось правильным.

## Хорошо

```go
func TestValidateEmailRejectsEmptyValue(t *testing.T) {
    // arrange
    // act
    // assert expected behavior
}
```

## Плохо

```go
func TestInternalStep3(t *testing.T) {
    // тест знает слишком много о внутреннем алгоритме
}
```

## Ключевые правила

- Название теста должно говорить, какое поведение проверяется.
- Проверяй успешные и ошибочные сценарии.
- Для похожих кейсов используй table-driven tests.
- Не делай тесты зависимыми от порядка выполнения.
- Тест должен быть воспроизводимым.

## Когда можно нарушить

- Для сложного алгоритма можно тестировать внутренние функции, если они имеют стабильный смысл.
- Для legacy-кода иногда сначала пишут characterization tests, которые фиксируют текущее поведение.
