# Project Structure

## Что это

Структура проекта — это организация папок и пакетов так, чтобы код было удобно запускать, тестировать и развивать.

## Минимальная учебная структура

```text
project/
├── go.mod
└── main.go
```

Этого достаточно для первых лабораторных.

## Более практичная структура

```text
project/
├── cmd/
│   └── app/
│       └── main.go
├── internal/
│   └── service/
├── go.mod
└── README.md
```

`cmd` хранит точки входа. `internal` хранит код, который не предназначен для импорта из других модулей.

## Ключевые правила

- Начинай просто, усложняй структуру только при реальной необходимости.
- Не копируй большую production-структуру для маленькой лабораторной.
- `main.go` должен запускать приложение, а не содержать всю бизнес-логику.
- Названия пакетов должны быть короткими и понятными.
- В одной папке должен быть один package.

## Внешний ориентир: project-layout

Репозиторий `golang-standards/project-layout` полезен как справочник распространенных директорий в Go application-проектах: `cmd`, `internal`, `pkg`, `api`, `configs`, `scripts`, `deployments` и других.

Важно не воспринимать его как обязательный стандарт. Для учебной лабораторной часто достаточно `go.mod` и `main.go`. Структура из большого репозитория нужна тогда, когда в проекте действительно появились отдельные точки входа, слой приложения, конфиги, миграции, deployment-файлы или несколько сервисов.

Практическое правило:

- `cmd` — точки входа приложений;
- `internal` — приватный код приложения, который не должен импортироваться извне;
- `pkg` — только для кода, который реально предполагается как переиспользуемый пакет;
- `api` — OpenAPI, protobuf, JSON schema и другие контракты;
- `configs` — шаблоны конфигурации без реальных секретов.

## Частые ошибки

- создать много папок до появления реальной логики;
- смешать разные package в одной директории;
- держать весь проект в одном `main.go`;
- использовать `pkg` без ясной причины;
- складывать временные бинарники и кэши в репозиторий.
